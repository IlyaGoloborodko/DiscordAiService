# CLAUDE.md

Guidance for Claude Code working in this repo. See [README.MD](README.MD) for the
full architecture, setup, and endpoint reference — this file is only the
non-obvious rules and gotchas.

## What this is

The AI decision layer (FastAPI + pydantic-ai) for a Discord music bot. Takes a
user message → decides an action → finds real tracks via a search microservice →
returns a reply + `tool_calls`. Local LLM over an OpenAI-compatible endpoint
(LM Studio). Entry point: `POST /agent` → `AgentService.run` in
[app/services/agent_service.py](app/services/agent_service.py).

## Hard constraints

- **Do NOT modify the sibling projects.** They live outside this repo and are
  read-only reference:
  - `discordAudio` (Go Discord bot) — captures voice, plays audio, does TTS.
  - `DsBotSearchService` (`:9000`) — yt-dlp/Last.fm search. Don't review or
    implement there unless explicitly asked.
- **Use official scaffolding commands** for anything meant to be generated — e.g.
  `alembic init`, `alembic revision --autogenerate`. Never hand-write files a tool
  is supposed to produce.
- **Keep the bot's contract flat.** The LLM output schema is deliberately flat
  (`action` + a flat `track_ids: list[str]`, never nested `tracks:[{...}]`). Local
  models' constrained decoders can't reliably emit arrays of objects, and the flat
  shape also lets the service drop hallucinated ids. Don't "clean it up" into a
  nested schema.

## Key design decisions (don't regress these)

- **System prompt is injected fresh into `message_history` every run**, not set on
  the `Agent`. pydantic-ai does not re-add a system prompt when `message_history`
  is non-empty, and trimming can drop stored system parts — so the service owns it.
- **Memory versioning:** `_MEMORY_VERSION` prefixes the session key. Bump it
  whenever the response contract or model changes, so stale history can't poison
  the model as few-shot examples. (Currently `v6`, after the qwen3.5 switch.)
- **Memory stores only clean conversational turns** — a `UserPromptPart` request +
  a `TextPart` response (Marina's `display_text`) — **never the raw agentic
  transcript** (tool calls, tool returns, reasoning). Replaying that scaffolding is
  fragile: strict chat templates (qwen) reject a history that doesn't start with the
  system message or has an orphaned tool-return, and it bloats context with search
  JSON. Don't go back to persisting `result.all_messages()`.
- **Only "clean" turns are persisted** (`_build_tool_calls` returns `clean`), so a
  bad/hallucinated answer never poisons the next turn.
- **`ModelHTTPError` retry loop** wraps `agent.run` — LM Studio can return a 400
  when its parser rejects its own output; pydantic-ai's `retries` only covers
  output validation, not HTTP errors.
- **Language & identity are env-templated in the prompt** via `{LANG}`, `{NAME}`,
  `{NAME_FORMS}` placeholders (`BOT_LANGUAGE`, `BOT_NAME`, `BOT_NAME_FORMS`).
  `display_text` must be in `BOT_LANGUAGE` because it feeds a single-language TTS
  voice; the spoken form is derived server-side (`clean_for_tts`), not generated by
  the model.
- **Env is read at request time** (`os.getenv` inside `_model` / `_system_text`),
  because `load_dotenv()` runs after import. Tests patch `os.environ` accordingly.

## Commands

```bash
uv sync                                      # install deps (uv, not pip)
uv run python main.py                         # run service on :8066
uv run python -m unittest test_agent_contract -v   # unit tests (no live model)
uv run alembic upgrade head                   # apply migrations
uv run alembic revision --autogenerate -m "msg"    # new migration
docker compose up -d redis                    # Redis hot cache
```

## Environment

- Windows / PowerShell is the primary shell (a Bash tool is also available).
- Python **3.14+**, dependency management via **uv**.
- Requires a running LLM endpoint (LM Studio), Redis, Postgres, and
  DsBotSearchService (`:9000`) for full live behavior. Unit tests need none of them.
